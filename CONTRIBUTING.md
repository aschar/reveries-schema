# Contributing

This is a small methodology, built by one person, offered publicly because the questions it tries to answer are bigger than any single project. Contributions are welcome in that spirit: serious, unhurried, grounded in actual experience with dreams.

## Before you open an issue

Read the Home page if you haven't. The schema has a particular intellectual lineage and a particular set of commitments. If those commitments don't make sense to you, the contributing guide probably isn't the right entry point. If they do, you are likely already noticing things that could be improved.

The most useful contributions come from people who have worked with the schema against real dreams, their own or a corpus they are studying, and found a place where it does not hold. A field that is ambiguous in practice. A symbol that keeps appearing without a natural home in the vocabulary. A confidence tier that seems miscalibrated for a particular class of dream. These are the observations worth bringing.

## Three ways to contribute

### Propose a new symbol

The current vocabulary has 16 slugs. It was built from one corpus and one analyst's experience, and it will have gaps. If you encounter dreams that don't map cleanly to any existing slug, and you have encountered enough of them to be confident this is a gap rather than an edge case, open a GitHub Issue with:

- The proposed symbol name and slug
- Three to five example dreams where it would apply
- Why existing slugs don't cover it
- Any Jungian amplification material that supports the addition

Symbol proposals are discussed in the Schema Room inside the Rêveries app before becoming formal PRs. If you don't have access to the app yet, open the issue anyway and the discussion can happen there.

### Propose a field refinement

If an existing field is ambiguous, inconsistently applied, or would be more useful in a different form, open a GitHub Issue with:

- The field name and its current definition
- The proposed change and the reasoning behind it
- The schema impact: does this require a migration, a backfill, a change to the confidence tier?

Field refinements touch existing data and need more careful handling than symbol additions. The issue is the right place to think through the implications before anything is written.

### Report a classification error

If you find a dream in the corpus where the enrichment is clearly wrong, a field that misidentified an archetype, a motif that was missed, a confidence tier that seems wrong for this class of content, open an issue with the dream ID and your analysis. No fix needed from you, just the observation.

## What happens after you open an issue

Issues are read. Some will open into longer discussions, in the GitHub thread or in the Schema Room. Those that reach a point of clarity become formal proposals. The maintainer makes the final decision on what merges.

This is not a democracy. That is worth saying plainly. The schema has philosophical commitments and a maintainer responsible for holding them. What the process offers is transparency: every decision is documented and reasoned, and anyone can see why something was accepted or declined.

## What will not be merged

Changes that flatten Tier 3 fields into Tier 1 certainty. Symbol additions that are culturally specific without cross-cultural amplification support. Structural changes proposed without schema impact analysis. Any change that removes the epistemic tier from a field. Changes that make the schema more efficient at the cost of making it less honest.

## A note on pace

This project moves at the pace of depth, not velocity. Issues may sit for a while before a response arrives. That is not neglect. It is the same unhurried quality the platform tries to bring to dreams themselves. If something is genuinely urgent, say so in the issue and it will be treated accordingly.
